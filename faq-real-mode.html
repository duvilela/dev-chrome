Por que o consumo no Modo Real é menor do que no Balão do Chrome?

Esclarecimentos sobre os bastidores da captura física de hardware pela API chrome.processes no Chrome Dev/Canary.

O que acontecia no Modo Simulação

No Chrome Estável, o StopRAM opera obrigatoriamente no Modo Simulação. Como as APIs de hardware são bloqueadas, o sistema gera consumos fictícios usando algoritmos de flutuação dinâmica.

Essa simulação foi calibrada com números relativamente altos (ex: `257 MB` ou `326 MB`) e flutuações constantes para fins estéticos e demonstrativos, simulando o peso que um site costuma ter. Isso gerava a impressão de que a extensão capturava o consumo real, mas eram valores puramente matemáticos.

A Arquitetura Multiprocesso do Chrome

Ao migrar para versões de desenvolvimento como o **Chrome Dev ou Canary**, a extensão ganha acesso de hardware e desativa a simulação, ativando o **Modo Real** (indicado pelo badge verde no painel).

Sob este modo, a diferença drástica entre os valores medidos pelo balão nativo do Chrome e o painel StopRAM se deve a como a memória é estruturada internamente no navegador:

  • O Balão do Chrome (Ex: 635 MB ou 842 MB): Mostra o consumo total acumulado em todas as pontas necessárias para rodar aquela aba. Isso engloba o processo do renderizador, scripts secundários, anúncios de terceiros em iframes, e principalmente o processo compartilhado de GPU (placa de vídeo) e o serviço de áudio, onde ficam os buffers pesados de decodificação de vídeo do YouTube.
  • A API chrome.processes (Ex: 30 MB): Por restrições de segurança de sandbox, a API oficial do Chrome só consegue mapear para a extensão a Memória Privada (Private Memory) exclusiva do processo renderizador principal da aba (a estrutura DOM, o código JavaScript ativo da página e estilos). A memória pesada de placa de vídeo e reprodução de áudio, que roda em processos compartilhados globais do navegador, não pode ser atribuída isoladamente a uma única aba pela API.

Agregação de Processos Ativada

Muitas abas modernas (especialmente sites complexos e com publicidade) criam processos renderizadores adicionais, conhecidos como sub-frames ou out-of-process iframes.

Para atenuar e aproximar os dados ao consumo máximo real, a lógica do StopRAM em background.js foi atualizada para **somar (acumular) a memória privada e o uso de CPU de todos os sub-processos associados ao mesmo ID de aba**. Isso consolida os dados de sub-frames secundários no valor final exibido no painel no Modo Real.

Como Configurar Limites no Modo Real

Como a API chrome.processes captura estritamente o consumo de memória privada do renderizador (que é bem menor que o consumo total da aba com a GPU), você deve ajustar a sua régua de limites no painel:

Dica de Calibração: No Modo Real, reduza o controle deslizante de limite de RAM de volta a valores entre 30 MB e 80 MB. Como os processos de renderização puros são leves, limites altos de 300 MB nunca serão atingidos e a aba não será suspensa. Configurando a régua mais baixa, você capturará picos no processamento da página e a suspensão atuará perfeitamente.
