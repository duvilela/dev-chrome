Funcionamento Técnico em Navegadores Estáveis

Esclarecimentos detalhados sobre o Modo Simulação e a coleta de dados de RAM e CPU no Chrome, Edge e Brave.

Restrição da API chrome.processes

Por questões rígidas de segurança e isolamento de processos (sandbox), o Google Chrome e demais navegadores baseados no Chromium (como Microsoft Edge, Brave e Opera) em suas versões estáveis bloqueiam o uso da API chrome.processes por extensões tradicionais.

Essa API fornece informações físicas sobre o Process ID (PID) e o consumo de hardware real de cada aba. Sem ela, uma extensão comum de navegador não possui permissões no sistema operacional para questionar o consumo físico individual de RAM de outra aba.

Nota: Para ativar o monitoramento físico em tempo real por meio da API chrome.processes, a extensão precisa ser executada em canais de desenvolvimento do navegador, como o Chrome Dev ou Chrome Canary.

Como Funciona o Modo Simulação

Quando detecta que a API do sistema está indisponível, o StopRAM ativa automaticamente seu motor híbrido de simulação no script de segundo plano (service worker). Ele simula o consumo de recursos das abas através de algoritmos de flutuação altamente realistas:

  • Consumo Inicial Diferenciado: Abas ativas (em foco) iniciam consumindo entre 120 MB e 270 MB. Abas de segundo plano iniciam em consumo reduzido, entre 40 MB e 140 MB.
  • Flutuação Dinâmica (Drift): A cada ciclo de verificação, o algoritmo altera de forma flutuante a RAM de cada aba. Abas ativas tendem a subir de consumo mais rapidamente devido à interação simulada do usuário.
  • Simulação de Vazamento de Memória (Memory Leak): A cada 7 abas criadas, o script ativa um multiplicador de consumo de 1.5x em uma delas. Isso faz com que essa aba cresça continuamente de forma artificial, simulando uma página mal otimizada até que o limite de RAM configurado seja atingido.
  • Uso de CPU e PID Coerentes: Abas ativas simulam o uso de CPU entre 4.0% e 18.0%, enquanto abas de fundo ficam abaixo de 1.5%. Também são criados Process IDs fictícios na faixa de 20000 + tabId.

De Onde os Dados São Puxados?

O StopRAM não utiliza APIs de hardware do sistema (como chrome.system.memory ou chrome.system.cpu) para calcular o consumo de abas individuais pelos seguintes motivos:

  • A API chrome.system.memory informa apenas dados globais da máquina (RAM total física e RAM livre geral no computador), não oferecendo detalhes de abas individuais.
  • A API chrome.system.cpu reporta apenas o uso e arquitetura geral dos núcleos físicos do processador.

Dessa forma, o StopRAM puxa os metadados reais das abas (Título, URL, Ícone, se está Fixada, Ativa ou Suspensa) diretamente da API chrome.tabs. Com base nessas abas reais identificadas, o script gera e calcula as métricas de RAM/CPU programaticamente utilizando matemática probabilística e persistência local no chrome.storage.local.

O Descarte das Abas é Real!

É fundamental ressaltar que, embora as métricas de RAM/CPU no Chrome estável sejam simuladas para testar e operar a extensão, **a ação de descarte das abas é 100% real**.

Quando a extensão atinge a regra limite e decide aplicar a ação de "Suspender Aba", ela executa a chamada nativa chrome.tabs.discard(tabId). Esta função é totalmente suportada em qualquer canal do Chrome e faz com que o navegador elimine fisicamente o processo daquela aba da memória RAM do seu computador de verdade, poupando hardware real.
